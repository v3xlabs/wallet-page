# Connect

A dapp needs an EIP-1193 `provider` and an account. Discovery comes first; `eth_requestAccounts` comes after the user picks a wallet. wallet.page keeps one provider + account for the other specification pages.

## What dapps should avoid

- **Assuming `window.ethereum`** — Only one injected provider wins; users with several extensions may never see the wallet they want.
- **Connecting on every page** — Hold one provider + account pair for the whole origin (this site does that via shared session state).
- **Hiding errors** — "User rejected" vs "no wallet installed" need different copy and next steps.
- **Blocking the UI** — Show connection state inline; don't trap users in modals with no dismiss path.

## Connection transports

**WalletConnect** and similar relay products route sessions through hosted infrastructure. We generally do not recommend them as the default desktop path when a local extension or an open protocol is available.

**[openlv](https://openlv.sh)** is open source and appears in the provider list above as a peer-to-peer EIP-1193 provider (not `window.ethereum`, not a vendor cloud relay).

## Wallet discovery

**[EIP-6963: Multi Injected Provider Discovery](https://eips.ethereum.org/EIPS/eip-6963)** avoids fighting over `window.ethereum`. Wallets announce themselves with two window events:

1. The dapp registers `window.addEventListener("eip6963:announceProvider", …)` and stores `event.detail` (`{ info, provider }`).
2. The dapp calls `window.dispatchEvent(new Event("eip6963:requestProvider"))` so wallets that loaded earlier announce again.

That is the full discovery loop. The connect panel above runs it on load; use **Request providers** to dispatch the request event again and expand **Announce payloads** to inspect each `info` object (`name`, `icon`, `rdns`, `uuid`).

:::note
**For wallet developers:** announce on load, listen for `eip6963:requestProvider`, and re-dispatch your `eip6963:announceProvider` event — as in the [reference implementation](https://eips.ethereum.org/EIPS/eip-6963#reference-implementation).
:::
